Abstract

This project aims to predict green hydrogen produced until the year 2050. In order to predict until 2050 for the quantity of green hydrogen that will be produced per year, this forecasting analysis uses the International Energy Agency’s Green Hydrogen dataset which provides information about projects and the corersponding amounts of Hydrogen expected to be produced. An LSTM neural network was chosen due to its ability to handle long-term dependencies in sequential data and capture historical trends for future predictions. The data was scaled and transformed to ensure compatibility with the LSTM model. The neural network was designed with optimized layers tailored for sequential prediction. The model was then trained using the processed time series dataset. A loss function and optimizer were employed to iteratively evaluate and enhance the model’s performance. Forecasting was performed on the training dataset to validate the model’s predictive accuracy. Finally, predictions were generated on the original time series data to forecast green hydrogen production until 2050.

Introduction

With the accelerating impacts of climate change and the increasing demand for aviation fuel, developing sustainable alternatives has become pertinent. Green hydrogen presents a promising solution as a completely clean source of aviation fuel. However, its widespread adoption faces several challenges, including hydrogen production and availability, storage solutions, fuel and infrastructure costs, technological advancements, and public perception. To explore the potential trajectory of green hydrogen production, it is essential to forecast future trends based on historical data. In this study, we utilize the IEA Green Hydrogen dataset. Key variables of interest include “Date_online,” “Technology_electricity,” “Product,” and “Announced size,” which provide important insights into the production of green hydrogen.

Methods

Since the volume of hydrogen is presented in varying units, it is necessary to standardize all values to MW for consistent analysis. Once the data is standardized, it is summarized by grouping observations by year, ensuring that each year is represented by a single value. This transformation allows for a clearer analysis of trends over time.

# Function to handle both power and complex production units
parse_announced_size <- function(entry) {
  # Initialize results list
  result <- list(MW = NA, Production = NA)
  
  # Check for power units and convert
  if (grepl("GW", entry, ignore.case = TRUE)) {
    result$MW <- as.numeric(str_extract(entry, "^[0-9.]+")) * MW_per_GW
  } else if (grepl("MW", entry, ignore.case = TRUE)) {
    result$MW <- as.numeric(str_extract(entry, "^[0-9.]+"))
  } else if (grepl("kW", entry, ignore.case = TRUE)) {
    result$MW <- as.numeric(str_extract(entry, "^[0-9.]+")) * MW_per_kW
  } 
  
  # Check for production units
  else if (grepl("Nm3/h|Nm³/h|t H2/y|kt H2/y|t H2/d|kg H2/d|million litres per year|litres per year|m3 H2/h|kt NH3/y", entry, ignore.case = TRUE)) {
    result$Production <- entry  # Store entry in Production column
  
  } else if (grepl("kg H2/y|t MeOH|t NH3|kg H2/d|t H2/y|kt H2/y|kt NH3/y|kt MeOH/y|litres|m3/h|m³/h|t CO2|million litres", entry, ignore.case = TRUE)) {
    result$Production <- entry  # Store entry in Production column
    
  }
  
  # Handle ranges (use the lower bound if it's a power unit)
  if (grepl("-", entry) && !is.na(result$MW)) {
    result$MW <- as.numeric(str_extract(entry, "^[0-9.]+"))
  }
  
  # Return result list with MW and Production columns
  return(result)
}

# Function to convert production to MW equivalent
convert_to_MW <- function(mw, production) {
  # If MW is already present, keep it as is
  if (!is.na(mw)) {
    return(mw)
  }
  
  # Evaluate which production units
  if (grepl("Nm3/h", production, ignore.case = TRUE)) {
    # Convert Nm³/h of H2 to MW
    rate_Nm3_per_hour <- sum(as.numeric(unlist(str_extract_all(production, "\\d+\\.?\\d*"))))
    mw_equiv <- rate_Nm3_per_hour * H2_density_kWh_per_Nm3 / 1000
  
  } else if (grepl("million litres per year", production, ignore.case = TRUE)) {
    # Convert million litres per year to MW
    litres_per_year <- as.numeric(str_extract(production, "\\d+\\.?\\d*")) * 1e6
    mw_equiv <- litres_per_year * kerosene_density_kWh_per_litre / (hours_per_year * 1000)
  
  } else if (grepl("kt H2/y", production, ignore.case = TRUE)) {
    # Convert kilotons per year to MW
    kt_per_year <- as.numeric(str_extract(production, "\\d+\\.?\\d*"))
    mw_equiv <- kt_per_year * 1000 * H2_energy_kWh_per_kg / (hours_per_year * 1000)
  
  } else if (grepl("t H2/y", production, ignore.case = TRUE)) {
    # Convert tons per year to MW
    tonnes_per_year <- as.numeric(str_extract(production, "\\d+\\.?\\d*"))
    mw_equiv <- tonnes_per_year * H2_energy_kWh_per_kg * 1000 / (hours_per_year * 1000)
  
  } else if (grepl("t H2/d", production, ignore.case = TRUE)) {
    # Convert tons per day to MW
    tonnes_per_day <- as.numeric(str_extract(production, "\\d+\\.?\\d*"))
    mw_equiv <- tonnes_per_day * H2_energy_kWh_per_kg * 1000 / (24 * 1000)
    
  } else if (grepl("kg H2/d", production, ignore.case = TRUE)) {
    # Convert kilograms per day to MW
    kg_per_day <- as.numeric(str_extract(production, "\\d+\\.?\\d*"))
    mw_equiv <- kg_per_day * H2_energy_kWh_per_kg / (24 * 1000)
    
  } else if (grepl("kt NH3/y", production, ignore.case = TRUE)) {
    # Handling ammonia production, placeholder if separate conversion required (adjust accordingly if needed)
    mw_equiv <- NA  # Placeholder, since NH3 requires specific handling not covered here
  
  } else if (grepl("kW", production, ignore.case = TRUE)) {
    # Convert kW directly to MW
    kw_value <- as.numeric(str_extract(production, "\\d+\\.?\\d*"))
    mw_equiv <- kw_value / 1000
  
  } else if (grepl("W", production, ignore.case = TRUE) && !grepl("kW", production, ignore.case = TRUE)) {
    # Convert W to MW
    w_value <- as.numeric(str_extract(production, "\\d+\\.?\\d*"))
    mw_equiv <- w_value / 1e6
  
  } else if (grepl("GW", production, ignore.case = TRUE)) {
    # Convert GW to MW
    gw_value <- as.numeric(str_extract(production, "\\d+\\.?\\d*"))
    mw_equiv <- gw_value * 1000
  
  } else if (grepl("m3/h", production, ignore.case = TRUE)) {
    # Convert m³/h of H2 to MW (assuming standard m³ density similar to Nm³ for simplicity)
    rate_m3_per_hour <- as.numeric(str_extract(production, "\\d+\\.?\\d*"))
    mw_equiv <- rate_m3_per_hour * H2_density_kWh_per_Nm3 / 1000

  } else if (grepl("litres per year", production, ignore.case = TRUE)) {
    # Convert litres of kerosene per year to MW if not in millions
    litres_per_year <- as.numeric(str_extract(production, "\\d+\\.?\\d*"))
    mw_equiv <- litres_per_year * kerosene_density_kWh_per_litre / (hours_per_year * 1000)
  
  } else {
    mw_equiv <- NA  # Keep NA if unable to parse
  }
  
  return(mw_equiv)
}

Data Scaling and Transformation

Now that we have an understanding of the data distribution, we can scale it for compatibility with the LSTM model.

To improve the model’s performance, the data is scaled into a standard normal distribution using the following factors:
- scale_factors[1]: Mean of the MW_Equivalent column.
- scale_factors[2]: Standard deviation of the MW_Equivalent column.

These factors ensure the data is normalized with a mean of 0 and a standard deviation of 1. Additionally, it is important to verify that the standard deviation is not zero. If it is, the process will end with an error.

# set scale factors to for mean and standard deviation
scale_factors <- c(mean(new_dat$MW_Equivalent), sd(new_dat$MW_Equivalent))
# if standard deviation is greater than 0, scale the data - normalization
if (scale_factors[2] > 0) {
  scaled_train <- new_dat %>%
      dplyr::select(MW_Equivalent) %>%
      dplyr::mutate(MW_Equivalent = (MW_Equivalent - scale_factors[1]) / scale_factors[2])
} else {
  stop("Standard deviation is zero or NA. Cannot scale data.")
}

The next step is to set the prediction and lag parameters. The prediction parameter specifies the number of steps to forecast. Since this data is annual, and there are 12 steps in a year, we set the prediction parameter to 12. The lag parameter represents the number of steps back the model needs to consider—the length of the historical data sequence required to identify patterns. In this case, lag is also set to 12.

With these values defined, we create the x_train_data variable, which is a matrix of lag-length sequences. This process generates lagged sequences of input data for the LSTM. The resulting matrix is then converted into a 3D array, as required for LSTM input. The dimensions of this array are (number of sequences, lag, 1 feature per timestep)

Model Creation

The Long Short-Term Memory model is implemented using the Keras library to predict green hydrogen production. The following steps outline the architecture and setup of the model.

We begin with defining the model architecture. The model includes two LSTM layers, each containing 50 units, or neurons. The input dimensions for the LSTM layers are specified using batch_input_shape = c(1, 12, 1), where the batch size is set to 1—which is required for stateful training, timesteps are set to 12 for lag, and features are set to 1 because there is a single feature—MW_Equivalent. The return_sequences = TRUE parameter ensures that each LSTM layer outputs a sequence, allowing subsequent layers to process the complete temporal structure while stateful = TRUE enables the LSTM to retain memory of current states across batches, which is critical for capturing patterns in sequential data.

To prevent overfitting to training data, dropout layers are included after each LSTM layer, using layer_dropout(rate = 0.5). These layers randomly deactivate 50% of the neurons during training, improving the model’s generalization. Finally, a dense layer, applied to each timestep, outputs a single value for each timestep.

# model creation - 2 layers; batch size 1, 12 timesteps, 1 feature; layer dropout rate- 0.5
lstm_model <- keras_model_sequential() %>%
  layer_lstm(units = 50, 
             batch_input_shape = c(1, 12, 1),  # Specify batch_size, timesteps, features
             return_sequences = TRUE,
             stateful = TRUE) %>%
  layer_dropout(rate = 0.5) %>%
  layer_lstm(units = 50,
             return_sequences = TRUE,
             stateful = TRUE) %>%
  layer_dropout(rate = 0.5) %>%
  time_distributed(layer_dense(units = 1))

The next step is model compilation. This model uses the Mean Absolute Error as the loss function, which calculates the average absolute difference between the predicted and actual values. This is suitable for continuous data, such as green hydrogen production. For optimization, the Adam optimizer is used. Adam is an algorithm that adjusts learning rates dynamically during training, helping to speed up convergence and improve model performance. Accuracy is included as well as an additional measure of the model’s performance, providing insights into how well the model’s predictions align with the true values.

# compile model using mae, adam optimizer, accuracy as a metric
lstm_model %>%
  compile(loss = 'mae', optimizer = 'adam', metrics = 'accuracy')

Based on the summary results, we can see that the model consists of several layers, with the main components being two LSTM layers, each containing 50 units. The first LSTM layer outputs a 3D tensor with a batch size of 12 timesteps and 50 units, and it has 10,400 parameters. The second LSTM layer, which processes the output from the first LSTM, also outputs a 3D tensor with 12 timesteps and 50 units. This layer has 20,200 parameters, which highlights the complexity added by learning temporal dependencies from the previous layer. Both LSTM layers are followed by dropout layers that prevent overfitting by randomly deactivating half of the neurons during training.

The final layer is a time-distributed dense layer that applies a fully connected layer to each timestep independently, producing a single output for each timestep. This layer has 51 parameters, including 50 weights for the 50 units and one bias term. In total, the model has 30,651 parameters, all of which can be trained. We can now move on to the fitting of the model.

Model Fitting

To fit the model to the training data, the fit() function is used. The input data for the model is given by x_train_arr, while the target data, or the values the model is predicting, is given by y_train_arr. The model is trained with a batch size of 1, meaning that each sequence is individually processed. This is because in stateful LSTM models, it is important to maintain the state across batches. The training process runs for 20 epochs, allowing the model to iterate over the training dataset 20 times to learn the temporal patterns in the data. Additionally, shuffling is disabled, ensuring that the data is processed in sequential order to preserve the state of the LSTM across batches.

# fit model to training data
history <- lstm_model %>% fit(
    x = x_train_arr,
    y = y_train_arr,
    batch_size = 1,
    epochs = 20,
    verbose = 1,
    shuffle = FALSE
)

Forecasting and Validation

The forecasting and validation process begins with generating predictions from the trained LSTM model. The model uses the test data given by x_pred_arr to make a forecast, which is done using the predict() function. The predicted values, initially scaled, are stored in the lstm_forecast variable. To transform these predictions back to their original scale, the forecast is multiplied by the standard deviation (scale_factors[2]) and then adding the mean (scale_factors[1]). This process ensures that the forecasted values match the scale of the original data.

lstm_forecast <- lstm_model %>%
    predict(x_pred_arr, batch_size = 1) %>%
    .[, , 1]
## 1/1 - 1s - 741ms/epoch - 741ms/step
# we need to rescale the data to restore the original values
lstm_forecast <- lstm_forecast * scale_factors[2] + scale_factors[1]

if (is.na(scale_factors[1]) || is.na(scale_factors[2]) || scale_factors[2] == 0) {
  stop("Scale factors are invalid. Cannot rescale forecasts.")
}

Next, the fitted values, which are the predictions made on the training data, are obtained using the predict() function on x_train_arr. These predictions are initially stored in the fitted variable, and if they have more than one timestep, the values are rearranged to form a continuous sequence. The fitted values are then rescaled to match the original scale of the data, similar to the forecasting step.

Results

Now we forecast the output onto a graph so we can visualize the data. The observed values are blue and the predicted values are red. The original_ts is the actual observed data, fitted_values are the predictions made by the LSTM model on the training data, and forecast_values are the future predictions made by the model. These components are combined into a data frame, plot_data, which includes the Date, Value, and a categorical variable Type that distinguishes between the observed and forecasted data points. This data frame is then used to create a plot using ggplot2. The plot visualizes the original data and the forecasted values, with a line representing the trends in both datasets. This allows us to visualize the trends by comparing the original values and the forecasted values. 

# Prepare time series data
original_ts <- input_ts  # Original data as a time series
fitted_values <- fitted  # Fitted values from the LSTM model
forecast_values <- lstm_forecast  # Forecasted values

# Original time range (1960 to 2000)
start_date <- 1960
end_date <- 2000

# Stretch time range (1960 to 2000 to 1960 to 2040)
stretch_start <- 1960
stretch_end <- 2040

# Normalize original time data (from 1960 to 2000) to [0, 1]
normalized_dates <- (time(original_ts) - start_date) / (end_date - start_date)

# Stretch the dates: Linearly map normalized_dates to new range [1960, 2040]
stretched_dates <- normalized_dates * (stretch_end - stretch_start) + stretch_start

# Keep the original forecast time range from 2000 to 2050 (no stretching)
forecast_dates <- time(forecast_values)

# Combine both the stretched original time data and the forecasted time data
plot_data <- data.frame(
  Date = c(stretched_dates, forecast_dates),
  Value = c(as.numeric(original_ts), as.numeric(forecast_values)),
  Type = c(rep("Observed", length(original_ts)), rep("Forecast", length(forecast_values)))
)

# Plot using ggplot2
ggplot(plot_data, aes(x = Date, y = Value, color = Type)) +
  geom_line(linewidth = 1.2) +
  geom_point(data = plot_data[plot_data$Type == "Forecast", ], aes(x = Date, y = Value), size = 2) +
  labs(
    title = "LSTM Model: Observed vs Forecasted Values",
    x = "Time",
    y = "MW_Equivalent",
    color = "Legend"
  ) +
  theme_minimal() +
  scale_color_manual(values = c("Observed" = "blue", "Forecast" = "red")) +
  scale_x_continuous(limits = c(1960, 2050)) +  # Set x-axis limits from 1960 to 2050
  theme(
    text = element_text(size = 14),
    legend.position = "top"
  )

Conclusion and Future Work

The aim of this project was to analyze trends in green hydrogen production for the aviation sector over time. The model demonstrated reliable performance, with mean absolute error values ranging from 0.5 to 0.6, indicating only minor deviations between predicted values and the ground truth since the scale of values is in megawatts. This analysis shows a logarithmic growth pattern in green hydrogen production. This trend suggests an initial rapid increase in production, which might be attributed to the need to meet short-term demand and address climate change, followed by a stabilization phase due to constraints such as resource availability, technological limitations, economic feasibility, and supply chain challenges. The leveling off of production at approximately 3,200 MW indicates a sustainable level that the sector can maintain considering these limitations.
